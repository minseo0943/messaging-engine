# ADR-004: 읽기 모델 저장소로 MongoDB 선택

## 상태
채택됨 (2026-04-06)

## 맥락
CQRS Query Side의 읽기 모델 저장소가 필요하다. 읽기 모델의 특성:
- 비정규화된 도큐먼트 (JOIN 없이 단일 조회)
- 유연한 스키마 (읽기 모델은 자주 변경됨)
- 고속 조회 + 페이지네이션

## 결정
**MongoDB를 CQRS 읽기 모델 저장소로 사용한다.**

## 이유
1. **비정규화 친화**: 채팅방 요약(ChatRoomView)처럼 중첩된 데이터를 하나의 도큐먼트로 저장
2. **스키마 유연성**: 읽기 모델 변경 시 마이그레이션 없이 필드 추가/변경 가능
3. **조회 성능**: 인덱스 + 비정규화로 단일 쿼리에 필요한 데이터를 모두 반환
4. **Spring Data MongoDB**: Repository 인터페이스만으로 CRUD + 페이지네이션 구현

## 대안 검토
| 대안 | 기각 이유 |
|------|----------|
| MySQL (읽기 전용 복제본) | 비정규화 모델을 관계형 스키마로 표현하면 복잡. 스키마 변경에 마이그레이션 필요 |
| Redis | 영속성 보장 부족, 복잡한 쿼리(페이지네이션, 필터링) 불가 |
| Elasticsearch 단독 | 쓰기 성능이 낮고, 실시간 반영에 refresh_interval 제약 |

## 결과
- `MessageDocument`: chatRoomId + createdAt 복합 인덱스로 히스토리 조회 최적화
- `ChatRoomView`: 채팅방당 하나의 도큐먼트로 메시지 수, 마지막 메시지 등 비정규화
- Kafka Consumer가 이벤트를 받아 MongoDB를 갱신 (Projection)
