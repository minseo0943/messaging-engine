# ADR-001: CQRS + 최종 일관성(Eventual Consistency) 선택

## 상태
채택됨 (2026-04-06)

## 맥락
메시징 서비스에서 쓰기(메시지 전송)와 읽기(히스토리 조회)는 요구사항이 매우 다르다:
- **쓰기**: 강한 정합성, 관계형 스키마, 트랜잭션
- **읽기**: 빠른 조회, 비정규화, 페이지네이션

단일 DB로 양쪽을 모두 만족시키면 읽기 성능을 위한 인덱스가 쓰기 성능을 저하시키고, 쓰기 스키마의 정규화가 읽기 쿼리를 복잡하게 만든다.

## 결정
**CQRS 패턴을 적용하여 Command(MySQL)와 Query(MongoDB)를 분리하고, Kafka 이벤트로 최종 일관성을 유지한다.**

## 이유
- **읽기/쓰기 독립 스케일링**: 읽기가 10배 많은 메시징 특성상 Query 서비스만 별도로 스케일아웃 가능
- **각 DB의 장점 활용**: MySQL(정규화, 트랜잭션) + MongoDB(비정규화, 빠른 읽기)
- **강한 일관성 포기의 현실적 타당성**: 채팅에서 "메시지 전송 후 0.5초 뒤 히스토리에 반영"은 사용자 경험에 영향 없음

## 대안 검토
| 대안 | 기각 이유 |
|------|----------|
| 단일 MySQL | 읽기 성능 한계, JOIN 쿼리 복잡도 |
| 강한 일관성 (2PC) | 성능 오버헤드, MSA에서 분산 트랜잭션은 안티패턴 |
| Change Data Capture (Debezium) | 인프라 복잡도 증가, 이 규모에서는 오버엔지니어링 |

## 결과
- 쓰기 응답시간: DB 저장 즉시 반환 (Kafka 발행은 비동기)
- 읽기 지연: Kafka → Consumer → MongoDB 프로젝션까지 평균 ~50ms
- 트레이드오프: 일시적으로 읽기 모델이 최신이 아닐 수 있음 (최종 일관성)
