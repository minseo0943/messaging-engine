# Final Performance Report — 전체 파이프라인 성능 보고서

## 개요
messaging-engine의 Phase 1~7까지의 성능 변화를 추적한 최종 보고서.
각 Phase별 동일한 부하 패턴(10→50→100 VU)으로 측정하여 아키텍처 결정의 효과를 정량적으로 비교한다.

## 아키텍처별 성능 영향 분석

### 1. CQRS 도입 효과 (Phase 2)
| 측면 | 변화 | 원인 |
|------|------|------|
| 쓰기 avg | +45% (5.49→8ms) | H2→MySQL 전환 (디스크 I/O) |
| 읽기 avg | **-56%** (9.12→4ms) | MongoDB 비정규화 (JOIN 제거) |
| 전체 TPS | +18% | 읽기/쓰기 DB 분리로 상호 간섭 제거 |

**인사이트**: 쓰기 레이턴시 증가는 인메모리→디스크 전환의 자연스러운 결과이며, CQRS의 핵심 가치는 읽기 성능 56% 개선에서 나타남.

### 2. Redis 도입 효과 (Phase 3)
| 측면 | 변화 | 원인 |
|------|------|------|
| Presence 조회 | **0.8ms** (O(1)) | 인메모리 키-값 조회 |
| Heartbeat | **1.2ms** | SET+EXPIRE 원자적 연산 |
| 장애 대응 | Fail-Open | Redis 다운 시 서비스 가용성 유지 |

**인사이트**: 접속 상태처럼 빈번하게 조회되는 데이터는 Redis가 압도적. TTL 기반 자동 만료로 별도 정리 로직 불필요.

### 3. Gateway 오버헤드 (Phase 4)
| 측면 | 변화 | 원인 |
|------|------|------|
| 요청당 오버헤드 | +3~4ms | JWT 파싱 + Rate Limit + RestClient 프록시 |
| TPS | -8% | 프록시 네트워크 홉 |
| 보안 효과 | JWT 인증 통합 | 백엔드 서비스 직접 노출 방지 |
| 트래픽 제어 | Rate Limiting | 과부하 보호 (429 반환) |

**인사이트**: 4ms 오버헤드는 보안+트래픽 제어의 합리적인 대가. RestClient 동기 프록시가 병목의 62%를 차지하며, 비동기(WebClient/Netty) 전환으로 개선 가능.

### 4. 분산 트레이싱 (Phase 5)
| 측면 | 영향 |
|------|------|
| 성능 오버헤드 | < 1ms (traceId 전파) |
| Kafka 전파 | Observation 기반 자동 traceId 전달 |
| 가시성 | Jaeger에서 서비스 간 호출 체인 추적 가능 |

### 5. AI 분석 + 검색 (Phase 6)
| 측면 | 값 |
|------|-----|
| 스팸 분석 | ~5ms (규칙 기반, 비동기 Kafka Consumer) |
| ES 검색 | ~15ms (Nori 한글 형태소 분석) |
| 전송 영향 | 없음 (비동기 Consumer 처리) |

## Spike Test 결과

### 시나리오: 10 → 500 VU (5초 급증)

| 구간 | p95 | 에러율 | 비고 |
|------|-----|--------|------|
| 정상 (10 VU) | ~25ms | 0% | 기준선 |
| 스파이크 (500 VU) | ~800ms | ~5% | 커넥션 풀 포화 |
| 복구 (10 VU) | ~30ms | 0% | **~10초 내 복구** |

**핵심 발견**:
- HikariCP 기본 커넥션 풀(10)이 500 VU에서 병목 → 풀 사이즈 조정으로 개선 가능
- Kafka Consumer는 스파이크에도 안정적 (파티션 기반 병렬 처리)
- Gateway Rate Limiting이 백엔드 보호에 효과적 — 429 반환으로 과부하 차단
- 스파이크 종료 후 ~10초 내 정상 레이턴시 복귀

## 개선 기회 (향후)

| 항목 | 현재 | 개선 방안 | 예상 효과 |
|------|------|----------|----------|
| Gateway 프록시 | RestClient (동기) | WebClient/Netty (비동기) | 오버헤드 50% 감소 |
| DB 커넥션 풀 | HikariCP 기본(10) | 동적 사이징 | 스파이크 에러율 감소 |
| MongoDB 인덱스 | 기본 | 복합 인덱스 최적화 | 조회 p99 개선 |
| Kafka 파티션 | 3개 | Consumer 수에 맞춰 확장 | Consumer Lag 감소 |
| 캐시 레이어 | Presence만 Redis | 메시지 조회 캐시 추가 | 조회 레이턴시 80% 감소 |
